## 2. Pre-Deployment Checklist

### 2.1 Database Migration

The v2 layer requires two new tables defined in the architecture (ARCH-20260210-0011-v2):

```sql
-- Migration: add_scheduled_projects
CREATE TABLE scheduled_projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(200) NOT NULL,
  description TEXT,
  delivery_date TIMESTAMPTZ NOT NULL,
  scheduled_start_date TIMESTAMPTZ,
  status VARCHAR(20) NOT NULL DEFAULT 'draft',
  priority VARCHAR(10) NOT NULL DEFAULT 'NORMAL',
  tags TEXT[] DEFAULT '{}',
  estimated_effort_hours FLOAT,
  actual_effort_hours FLOAT DEFAULT 0,
  created_by VARCHAR(100) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);

CREATE INDEX idx_scheduled_projects_status ON scheduled_projects(status);
CREATE INDEX idx_scheduled_projects_priority ON scheduled_projects(priority);
CREATE INDEX idx_scheduled_projects_delivery_date ON scheduled_projects(delivery_date);
CREATE INDEX idx_scheduled_projects_created_by ON scheduled_projects(created_by);

CREATE TABLE project_phases (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES scheduled_projects(id) ON DELETE CASCADE,
  phase VARCHAR(20) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  assigned_agent_id VARCHAR(100),
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  output_artifacts JSONB DEFAULT '[]',
  notes TEXT,
  duration_minutes INT,
  UNIQUE(project_id, phase)
);

CREATE INDEX idx_project_phases_status ON project_phases(status);
CREATE INDEX idx_project_phases_agent ON project_phases(assigned_agent_id);
```

**Steps**:
1. Add models to `prisma/schema.prisma`
2. Run `npx prisma migrate dev --name add_scheduled_projects`
3. Verify migration applies cleanly on staging database

## 3. Deployment Strategy

### Strategy: Rolling Deployment (Blue-Green for database)

**Rationale**: The v2 layer is purely additive -- no existing tables, endpoints, or components are modified. This makes a rolling deployment safe.

### Sequence

```
1. Database Migration (non-breaking, additive)
   - Apply Prisma migration to add new tables
   - Verify tables exist with correct schema
   - No downtime required (new tables only)

2. Backend Deployment (rolling)
   - Deploy new code with scheduling v2 module
   - New endpoints become available at /api/v1/scheduling/projects/*
   - Existing v1 endpoints unaffected

3. Frontend Deployment (rolling)
   - Deploy webapp with MultiProjectDashboard component
   - New route becomes accessible
   - Existing scheduling dashboard unaffected

4. Smoke Test
   - Create test project via API
   - Verify 7 phases initialize
   - Verify dashboard loads
   - Verify agent pool status displays

5. Feature Flag (Optional)
   - Enable multi-project feature for specific users first
   - Gradual rollout to all users
```

## 4. Rollback Plan

### Trigger Conditions
- API error rate > 5% on project endpoints
- Database migration failure
- Dashboard rendering errors affecting existing views

### Rollback Steps

1. **Frontend**: Revert to previous webapp build (v1 dashboard remains functional)
2. **Backend**: Revert to previous API build (v1 scheduling routes remain functional)
3. **Database**: New tables can be dropped without affecting existing data:
   ```sql
   DROP TABLE IF EXISTS project_phases;
   DROP TABLE IF EXISTS scheduled_projects;
   ```
   Apply Prisma migration rollback: `npx prisma migrate resolve --rolled-back [migration_name]`

### Recovery Time Estimate
- Frontend rollback: < 2 minutes
- Backend rollback: < 5 minutes
- Database rollback: < 5 minutes
