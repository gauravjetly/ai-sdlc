## Deployment Components

### 1. Database Migration

**New Tables Required**:

```sql
-- Migration: 20260210_add_multi_project_scheduling

-- ScheduledProjects table
CREATE TABLE scheduled_projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(200) NOT NULL,
  description TEXT,
  delivery_date TIMESTAMPTZ NOT NULL,
  scheduled_start_date TIMESTAMPTZ,
  status VARCHAR(20) NOT NULL DEFAULT 'draft',
  priority VARCHAR(10) NOT NULL DEFAULT 'NORMAL',
  tags TEXT[] DEFAULT '{}',
  estimated_effort_hours FLOAT,
  actual_effort_hours FLOAT DEFAULT 0,
  created_by VARCHAR(100) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);

-- Indexes for performance
CREATE INDEX idx_scheduled_projects_status ON scheduled_projects(status);
CREATE INDEX idx_scheduled_projects_priority ON scheduled_projects(priority);
CREATE INDEX idx_scheduled_projects_delivery_date ON scheduled_projects(delivery_date);
CREATE INDEX idx_scheduled_projects_created_by ON scheduled_projects(created_by);

-- ProjectPhases table
CREATE TABLE project_phases (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES scheduled_projects(id) ON DELETE CASCADE,
  phase VARCHAR(20) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  assigned_agent_id VARCHAR(100),
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  output_artifacts JSONB DEFAULT '[]',
  notes TEXT,
  duration_minutes INT,
  UNIQUE(project_id, phase)
);

-- Indexes for agent pool queries
CREATE INDEX idx_project_phases_status ON project_phases(status);
CREATE INDEX idx_project_phases_agent ON project_phases(assigned_agent_id);
CREATE INDEX idx_project_phases_project_id ON project_phases(project_id);
```

## Rollback Plan

### Trigger Conditions

- API error rate > 10% for 5 minutes
- Database migration failure
- Redis connection failures preventing critical operations
- Frontend UI completely broken (white screen, JS errors)
- Health checks failing after deployment

### Rollback Procedure

**Step 1: Revert ECS Services (2 minutes)**

```bash
# Rollback API service to previous task definition
aws ecs update-service \
  --cluster catalyst-dev-cluster \
  --service catalyst-dev-api \
  --task-definition catalyst-dev-api:PREVIOUS_REVISION \
  --force-new-deployment

# Rollback UI service
aws ecs update-service \
  --cluster catalyst-dev-cluster \
  --service catalyst-dev-ui \
  --task-definition catalyst-dev-ui:PREVIOUS_REVISION \
  --force-new-deployment
```

**Step 2: Rollback Database (5 minutes)**

```bash
# Connect to staging database
psql $DATABASE_URL

-- Drop new tables (safe - no data dependencies)
DROP TABLE IF EXISTS project_phases CASCADE;
DROP TABLE IF EXISTS scheduled_projects CASCADE;

-- Verify rollback
\dt scheduled_projects
-- Expected: relation does not exist
```

**Step 3: Verify Rollback (2 minutes)**

```bash
# Check API health
curl https://staging-alb-dns/health

# Check existing v1 scheduling still works
curl https://staging-alb-dns/api/v1/scheduling/work-items
```

**Step 4: Post-Rollback Actions**

- Document rollback reason
- Create incident report
- Schedule post-mortem
- Plan remediation deployment
