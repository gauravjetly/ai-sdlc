## 1. Refined Problem Statement

The v1 requirements defined scheduling at the **work item** level. The refined understanding elevates the unit of work to the **AISDLC project** level. Users need to schedule entire projects -- each with a full 7-phase lifecycle (BA, Architecture, Engineering, Security, QA, Deployment, Acceptance) -- and run multiple such projects concurrently. Agents are shared resources allocated across a pool of active projects, not dedicated to individual tasks.

### Key Conceptual Shift
| Concept | v1 (Work Items) | v2 (Multi-Project) |
|---------|-----------------|---------------------|
| Unit of scheduling | Single work item | Full SDLC project |
| Agent assignment | 1 agent per task | Agent pool across N projects |
| Lifecycle | Execute and done | 7-phase pipeline per project |
| Concurrency | Tasks run in parallel | Projects run in parallel, agents time-share |
| Memory | Per-agent context | Per-project and cross-project learning |

## 2. New Domain: ScheduledProject

A **ScheduledProject** is the primary entity. It wraps the full AISDLC lifecycle as a schedulable, trackable unit.

### ScheduledProject Properties
- **id**: UUID
- **name**: Human-readable project name (e.g., "Auth Service v2")
- **description**: Detailed project description / brief
- **deliveryDate**: Target delivery date
- **scheduledStartDate**: When the project should begin (nullable = immediate)
- **currentPhase**: Which SDLC phase is active (requirements | architecture | development | security | testing | deployment | acceptance)
- **phaseHistory**: Array of phase completion records with timestamps and outputs
- **status**: draft | scheduled | in_progress | blocked | completed | failed | cancelled
- **priority**: low | normal | high | critical
- **tags**: string[] for categorization and filtering
- **createdBy**: User who created the project
- **estimatedEffortHours**: Estimated total effort
- **actualEffortHours**: Tracked actual effort

### Phase Model (per project)
Each project has 7 phases, each with:
- **phase**: Enum (requirements, architecture, development, security, testing, deployment, acceptance)
- **status**: pending | in_progress | completed | failed | blocked | skipped
- **assignedAgentId**: Which agent instance is handling this phase
- **startedAt, completedAt**: Timestamps
- **outputArtifacts**: JSON array of file paths and descriptions
- **notes**: Agent-generated notes and decisions

## 3. New Functional Requirements

### FR-011: Schedule Full SDLC Projects
**Priority**: P0
**Description**: Users can create and schedule complete SDLC projects from the UI. Each project includes a delivery date and automatically progresses through all 7 phases.

**Acceptance Criteria**:
- Given a user fills in project details (name, description, delivery date), When they submit, Then a ScheduledProject is created with 7 pending phases
- Given a project's start date arrives, When the scheduler triggers, Then the BA phase begins automatically
- Given the BA phase completes, When its output is saved, Then the Architecture phase begins with the BA output as input

### FR-012: Concurrent Multi-Project Execution
**Priority**: P0
**Description**: Multiple projects can be in progress simultaneously. The system orchestrates agent resources across all active projects based on priority and delivery deadlines.

**Acceptance Criteria**:
- Given 5 projects are scheduled, When 3 of them reach their start date, Then all 3 begin their first phase concurrently
- Given an agent is working on Project A's development phase, When Project B (higher priority) reaches development, Then the system allocates a second agent instance or queues Project B's development with priority
- Given agent capacity is reached, When a new project starts, Then it is queued with an ETA displayed to the user

### FR-013: Agent Pool Management
**Priority**: P0
**Description**: Agents are managed as a shared pool. The system tracks which agents are available, busy, or idle and allocates them to project phases intelligently.

**Acceptance Criteria**:
- Given the agent pool has 3 developer agents, When 5 projects need development simultaneously, Then the system queues 2 projects with ETAs and processes 3 concurrently
- Given an agent completes a phase for one project, When it becomes free, Then it is automatically assigned to the highest-priority waiting phase
- Given the dashboard is open, When agent allocation changes, Then the pool status updates in real time

### FR-014: Cross-Project Agent Memory
**Priority**: P1
**Description**: Agents accumulate knowledge across projects. An architect agent that designed a microservice architecture retains those patterns for subsequent projects.

**Acceptance Criteria**:
- Given an architect agent completed Project A's architecture using event-driven patterns, When it begins Project B's architecture, Then it can recall and reuse those patterns
- Given agent memory contains decisions from 10 prior projects, When a new similar project starts, Then the agent suggests reusable patterns from memory

### FR-015: Project Pipeline Visualization
**Priority**: P0
**Description**: The dashboard shows all projects as pipelines, with each phase as a stage. Users see at a glance which phase every project is in.

**Acceptance Criteria**:
- Given 8 active projects, When the user views the dashboard, Then they see all 8 as compact pipeline rows showing 7 phase indicators each
- Given a project is in the security phase, When the user looks at that row, Then phases 1-3 show as completed (green), phase 4 shows as active (blue), phases 5-7 show as pending (gray)

### FR-016: Delivery Date Intelligence
**Priority**: P1
**Description**: The system calculates whether projects are on track to meet their delivery dates based on current phase and historical velocity.

**Acceptance Criteria**:
- Given a project has a delivery date of Feb 20 and is currently in architecture (phase 2 of 7), When the system calculates average phase duration from completed projects, Then it shows "on track" or "at risk" or "behind" status
- Given a project is behind schedule, When the dashboard renders, Then it shows a visual warning with the estimated actual completion date

## 4. Updated Data Model

### ScheduledProject (NEW entity)
```
ScheduledProject {
  id                  UUID        PK
  name                String      required
  description         String      optional
  deliveryDate        DateTime    required
  scheduledStartDate  DateTime    nullable (null = immediate)
  currentPhase        SDLCPhase   enum
  status              ProjectStatus enum
  priority            Priority    enum
  tags                String[]
  estimatedEffortHours Float      optional
  actualEffortHours    Float      default 0
  createdBy           String      required
  createdAt           DateTime
  updatedAt           DateTime
  completedAt         DateTime    nullable
}
```

### ProjectPhase (NEW entity)
```
ProjectPhase {
  id                  UUID        PK
  projectId           UUID        FK -> ScheduledProject
  phase               SDLCPhase   enum (requirements|architecture|development|security|testing|deployment|acceptance)
  status              PhaseStatus enum (pending|in_progress|completed|failed|blocked|skipped)
  assignedAgentId     String      nullable
  startedAt           DateTime    nullable
  completedAt         DateTime    nullable
  outputArtifacts     JSON        array of {name, path, type}
  inputFromPhase      UUID        nullable FK -> ProjectPhase (previous phase)
  notes               String      optional
  durationMinutes     Int         nullable
}
```

### AgentPoolStatus (runtime, not persisted)
```
AgentPoolStatus {
  agentType           String
  totalInstances      Int
  busyInstances       Int
  idleInstances       Int
  queuedPhases        Int         phases waiting for this agent type
  avgPhaseDurationMin Float       historical average
}
```

### SDLCPhase Enum
```
requirements | architecture | development | security | testing | deployment | acceptance
```

### ProjectStatus Enum
```
draft | scheduled | in_progress | blocked | completed | failed | cancelled
```

## 6. Dashboard Analytics Requirements (Compact, Modern, Insightful)

### Key Metrics (visible at a glance)
1. **Active Projects**: Count with phase distribution sparkline
2. **Delivery Health**: Pie/donut showing on-track vs at-risk vs behind
3. **Agent Utilization**: Compact bar showing busy/idle per agent type
4. **Throughput**: Projects completed per week (trend line)
5. **Avg Phase Duration**: Per-phase timing with variance indicator
6. **Velocity Trend**: Improving, stable, or degrading over last 4 weeks

### Compact Design Principles
- No full-width cards -- use inline stat chips and sparklines
- Pipeline view as horizontal phase indicators (7 dots per project)
- Agent pool as a single horizontal stacked bar
- Use color intensity rather than separate components to show status
- Maximum information in minimum vertical space

### Acceptance Criteria for Dashboard
- Given the user opens the dashboard, When 15 projects exist, Then the entire portfolio is visible without scrolling (all projects + metrics in viewport)
- Given the dashboard loads, When the user glances at it for 3 seconds, Then they can identify: how many projects are active, which are at risk, and agent utilization level
- Given a project changes phase, When the update arrives via WebSocket, Then the dashboard updates the pipeline indicator within 1 second
