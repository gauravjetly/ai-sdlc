## 1. Problem Statement

The current AI-SDLC Agent Mesh operates with file-based communication and keyword-based knowledge search. This architecture cannot support the semantic retrieval, real-time event-driven communication, and scalable coordination required for a true agentic AI platform. Phase 1 upgrades the foundation by adding PostgreSQL with pgvector for semantic memory and Redis with BullMQ for event-driven messaging, while maintaining full backward compatibility with the existing file-based system.

## 2. Stakeholders

| Stakeholder | Role | Concern |
|---|---|---|
| Platform Engineers | Build and maintain the platform | Clean abstractions, testability, migration safety |
| Agent Developers | Create and configure agents | Agents should work with both old and new infrastructure |
| End Users | Use /sdlc-* commands | Zero disruption to existing workflow |
| Operations | Deploy and monitor | Docker Compose for local, configuration toggles |

## 3. Scope

### In Scope
- PostgreSQL 15 with pgvector extension setup and schema
- Redis 7 with BullMQ for event-driven messaging
- MemoryService abstraction layer (provider pattern)
- EventBus abstraction layer (provider pattern)
- Semantic vector search replacing calculateRelevance()
- Hybrid search (vector + keyword with Reciprocal Rank Fusion)
- Embedding generation service (OpenAI text-embedding-3-large)
- Dual-write strategy (file + database) during migration
- Configuration toggles for storage and event bus providers
- Migration of existing collective memory data
- Docker Compose setup for local development
- Regression test suite for existing Agent Mesh functionality

### Out of Scope
- Reasoning Engine (Phase 2)
- Goal Decomposition and DAG Planner (Phase 2)
- Policy Engine (Phase 2)
- MCP Tool Layer (Phase 3)
- Production Kubernetes deployment (Phase 4)
- UI/UX changes (no frontend in this phase)

## 4. Functional Requirements

### FR-001: PostgreSQL Database Setup
**Priority**: P0 (Critical)
**Description**: Set up PostgreSQL 15 with pgvector extension, create the database schema per ARCH doc Section 7.1.

**Acceptance Criteria**:
- Given a fresh development environment
- When Docker Compose is started
- Then PostgreSQL 15 is running with pgvector extension enabled
- And all tables are created (agents, workflows, tasks, task_dependencies, knowledge, episodes, events, policies, audit_log)
- And IVFFlat index is created on knowledge.embedding column
- And GIN index is created on knowledge.tags column

### FR-002: Database Connection Pool and Migration System
**Priority**: P0 (Critical)
**Description**: Implement connection pooling and a database migration system for schema versioning.

**Acceptance Criteria**:
- Given the application starts
- When connecting to PostgreSQL
- Then a connection pool is established with configurable pool size
- And pending migrations are applied automatically
- And migration history is tracked
- And connection errors are handled gracefully with retries

### FR-003: MemoryService Abstraction Layer
**Priority**: P0 (Critical)
**Description**: Create a MemoryService interface with provider pattern that supports both file-based and PostgreSQL-backed storage.

**Acceptance Criteria**:
- Given the MemoryService interface is defined
- When STORAGE=postgres is configured
- Then all memory operations use PostgreSQL
- When STORAGE=file is configured
- Then all memory operations use the existing file-based system
- And the interface includes: storeKnowledge, searchKnowledge, getKnowledge, addEvidence, deprecateKnowledge

### FR-004: Embedding Generation Service
**Priority**: P0 (Critical)
**Description**: Implement a service that generates vector embeddings using OpenAI's text-embedding-3-large model.

**Acceptance Criteria**:
- Given text content to embed
- When the embedding service is called
- Then a 3072-dimensional vector is returned
- And batch processing supports up to 100 items per batch
- And API errors are handled with exponential backoff retry
- And embedding results are cached to avoid duplicate API calls
- And a fallback to text-embedding-3-small (1536 dimensions) is available

### FR-005: Vector Search with pgvector
**Priority**: P0 (Critical)
**Description**: Implement vector similarity search using pgvector cosine distance.

**Acceptance Criteria**:
- Given knowledge items stored with embeddings
- When a semantic search query is executed
- Then the top-K most semantically similar results are returned
- And results are filtered by optional metadata (category, agent, confidence, date range)
- And minimum similarity threshold is enforced
- And search completes in under 50ms for up to 10,000 knowledge items

### FR-006: Hybrid Search (Vector + Keyword with RRF)
**Priority**: P1 (High)
**Description**: Implement hybrid retrieval combining vector similarity and keyword search using Reciprocal Rank Fusion.

**Acceptance Criteria**:
- Given a search query
- When hybrid search is executed
- Then vector search returns top-K candidates based on embedding similarity
- And keyword search returns top-K candidates based on text matching
- And Reciprocal Rank Fusion combines both result sets
- And searching "user login" finds knowledge titled "authentication flow"
- And searching exact identifiers (e.g., "FR-001") still returns exact matches

### FR-007: Replace calculateRelevance() with Vector Search
**Priority**: P1 (High)
**Description**: Replace the existing string-containment-based calculateRelevance() method in CollectiveMemory with the new vector-based search.

**Acceptance Criteria**:
- Given the STORAGE=postgres configuration
- When CollectiveMemory.search() is called with a query
- Then vector-based hybrid search is used instead of calculateRelevance()
- And all existing search behavior is preserved when STORAGE=file
- And relevance results are measurably better (>30% improvement in retrieval quality benchmark)

### FR-008: Redis Setup with BullMQ
**Priority**: P0 (Critical)
**Description**: Set up Redis 7 and implement BullMQ-based event bus.

**Acceptance Criteria**:
- Given Docker Compose is started
- When the application connects
- Then Redis 7 is available for BullMQ
- And BullMQ queues are created for each event topic
- And connection health is monitored

### FR-009: EventBus Abstraction Layer
**Priority**: P0 (Critical)
**Description**: Create an EventBus interface with provider pattern supporting BullMQ and file-based providers.

**Acceptance Criteria**:
- Given the EventBus interface is defined
- When EVENT_BUS=bullmq is configured
- Then events are published and consumed via BullMQ/Redis
- When EVENT_BUS=file is configured
- Then events use the existing file-based MessageBus
- And the interface includes: publish, subscribe, unsubscribe, replay, getHistory

### FR-010: BullMQ EventBus Provider
**Priority**: P0 (Critical)
**Description**: Implement the BullMQ-backed EventBus provider with topic hierarchy.

**Acceptance Criteria**:
- Given the BullMQ provider is configured
- When an event is published to a topic (e.g., "task.completed")
- Then all subscribers to that topic receive the event within 100ms
- And subscribers using wildcard patterns (e.g., "task.*") receive matching events
- And events are persisted to PostgreSQL for replay
- And failed event processing is retried with configurable backoff
- And dead-letter queue captures permanently failed events

### FR-011: File-Based EventBus Provider
**Priority**: P1 (High)
**Description**: Refactor the existing MessageBus into a file-based EventBus provider implementing the same interface.

**Acceptance Criteria**:
- Given the file provider is configured
- When events are published
- Then they are written to the existing file-based inbox structure
- And the existing MessageBus behavior is preserved
- And all existing Agent Mesh tests pass without modification

### FR-012: Topic Hierarchy and Subscription Patterns
**Priority**: P1 (High)
**Description**: Implement hierarchical topic routing per ADR-031.

**Acceptance Criteria**:
- Given the topic hierarchy (task.*, workflow.*, learning.*, conflict.*, agent.*, system.*)
- When an agent subscribes to "task.*"
- Then it receives events for task.created, task.completed, task.failed, etc.
- And topic-specific handlers are invoked correctly
- And subscription management (subscribe/unsubscribe) works correctly

### FR-013: Event Persistence and Replay
**Priority**: P2 (Medium)
**Description**: Persist all events to PostgreSQL and support replaying events by topic and time range.

**Acceptance Criteria**:
- Given events have been published
- When replay is requested for a topic and time range
- Then all matching events are returned in chronological order
- And trace IDs link related events across agents

### FR-014: Configuration Toggle System
**Priority**: P0 (Critical)
**Description**: Implement environment-based configuration toggles for selecting storage and event bus providers.

**Acceptance Criteria**:
- Given environment variables STORAGE and EVENT_BUS
- When STORAGE=postgres and EVENT_BUS=bullmq
- Then the application uses PostgreSQL and BullMQ
- When STORAGE=file and EVENT_BUS=file
- Then the application uses the existing file-based systems
- And default configuration uses file-based providers (backward compatible)
- And configuration can be set via .env file or environment variables

### FR-015: Data Migration
**Priority**: P1 (High)
**Description**: Create migration scripts to move existing collective memory data from files to PostgreSQL with embeddings.

**Acceptance Criteria**:
- Given existing collective memory files in ~/.claude/agent-mesh/collective-memory/
- When the migration script is executed
- Then all knowledge items are inserted into the PostgreSQL knowledge table
- And embeddings are generated for each knowledge item
- And the migration is idempotent (safe to run multiple times)
- And a migration report is generated showing items migrated

### FR-016: Docker Compose Development Environment
**Priority**: P0 (Critical)
**Description**: Create Docker Compose configuration for local development with PostgreSQL and Redis.

**Acceptance Criteria**:
- Given the project is cloned
- When `docker-compose up` is executed
- Then PostgreSQL 15 with pgvector is running on port 5432
- And Redis 7 is running on port 6379
- And health checks verify both services are ready
- And data is persisted across container restarts via volumes

## 5. Non-Functional Requirements

### NFR-001: Performance
- Vector search: < 50ms for top-5 results on 10,000 knowledge items
- Event delivery: < 100ms from publish to subscriber notification
- Database connection pool: 10-20 connections (configurable)
- Embedding generation: < 500ms per item (API latency)

### NFR-002: Backward Compatibility
- All existing /sdlc-* commands must work without modification
- All existing Agent Mesh tests must pass (regression prevention)
- File-based mode must be the default (opt-in to new infrastructure)
- No breaking changes to the AgentMesh public API

### NFR-003: Reliability
- Database connection failures must be handled gracefully with retries
- Redis connection failures must fall back to file-based messaging
- Embedding API failures must not block knowledge storage (embed later)
- Circuit breaker pattern retained for agent communication

### NFR-004: Security
- Database credentials stored in environment variables, never in code
- Redis password required in non-development environments
- No raw SQL queries; use parameterized queries only
- Embedding API key stored securely (environment variable)

### NFR-005: Testability
- All new components must have unit tests (>80% coverage)
- Integration tests for both PostgreSQL and file-based providers
- Regression test suite verifying all existing Agent Mesh behavior
- Mock providers for unit testing without external dependencies

### NFR-006: Observability
- Database query logging (configurable verbosity)
- Event bus metrics (publish rate, delivery latency, queue depth)
- Memory service operation logging
- Health check endpoints for PostgreSQL and Redis

### NFR-007: Scalability
- PostgreSQL schema supports up to 100,000 knowledge items
- BullMQ supports 5+ concurrent workflows
- Connection pooling prevents database exhaustion
- IVFFlat index supports efficient approximate nearest neighbor search

## 6. Technical Constraints

1. **Runtime**: Node.js 20+ with TypeScript 5.3+
2. **Database**: PostgreSQL 15 with pgvector 0.5+
3. **Cache/Queue**: Redis 7+
4. **Embedding Model**: OpenAI text-embedding-3-large (3072 dimensions)
5. **Queue Library**: BullMQ 5+
6. **PostgreSQL Client**: pg (node-postgres) with pg-pool
7. **Testing**: Jest 29+
8. **Container**: Docker Compose for local development

## 9. Risks and Mitigations

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Vector search quality is poor for code-related content | Medium | Medium | Hybrid search (vector + keyword); benchmark before replacing |
| Migration breaks existing workflows | Medium | High | Dual-write strategy; feature flags; comprehensive regression tests |
| Embedding API rate limits during migration | Low | Medium | Batch processing with rate limiting; retry with backoff |
| Redis unavailability disrupts messaging | Low | High | File-based fallback; graceful degradation |
| PostgreSQL pgvector index performance degrades at scale | Low | Medium | IVFFlat with appropriate list count; monitor query times |
