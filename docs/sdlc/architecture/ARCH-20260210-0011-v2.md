## 1. Architectural Evolution

The v1 architecture provided scheduling primitives (ScheduledWorkItem, triggers, agent memory). The v2 layer adds a **Project Orchestration** tier that models full SDLC projects as first-class entities and manages a shared agent pool to serve multiple concurrent projects.

### Layered View

```
+------------------------------------------------------------------+
|  Presentation: Dashboard, Project UI, Analytics                   |
+------------------------------------------------------------------+
|  Application: ProjectOrchestrationService, AgentPoolService       |  <-- NEW v2
+------------------------------------------------------------------+
|  Application: SchedulingService, TriggerService, MemoryService    |  <-- v1
+------------------------------------------------------------------+
|  Domain: ScheduledProject, ProjectPhase, AgentPool                |  <-- NEW v2
+------------------------------------------------------------------+
|  Domain: ScheduledWorkItem, WorkTrigger, AgentMemory              |  <-- v1
+------------------------------------------------------------------+
|  Infrastructure: Prisma, BullMQ, Redis, WebSocket                 |
+------------------------------------------------------------------+
```

## 3. Multi-Project Scheduling Algorithm

### Phase Advancement Flow

```
                 +---> [Requirements] ---> [Architecture] ---> [Development]
                 |
[Project Start]--+---> concurrent projects follow same pipeline
                 |
                 +---> [Security] ---> [Testing] ---> [Deployment] ---> [Acceptance]
```

### Agent Allocation Algorithm (Priority-Deadline Scheduling)

```
function allocateAgents():
  1. Get all phases with status = "pending" where previous phase = "completed"
  2. Sort by:
     a. Project priority (CRITICAL > HIGH > NORMAL > LOW)
     b. Delivery date (soonest first)
     c. Phase order (earlier phases first -- clear bottlenecks)
  3. For each pending phase:
     a. Check if agent of required type is available in pool
     b. If available: assign agent, start phase
     c. If not available: calculate ETA based on current busy agents' avg duration
     d. Update project with ETA
  4. Emit 'phase.started' or 'phase.queued' events
```

### Delivery Date Risk Calculation

```
function calculateDeliveryHealth(project):
  remainingPhases = phases where status in (pending, in_progress)
  avgPhaseDuration = historicalAvg(completedPhases across all projects)
  estimatedRemainingDays = sum(remainingPhases.map(p => avgDuration[p.phase]))
  estimatedCompletion = now + estimatedRemainingDays

  if estimatedCompletion <= deliveryDate:
    return "on_track"
  elif estimatedCompletion <= deliveryDate + 2 days:
    return "at_risk"
  else:
    return "behind"
```

## 5. Data Model Extensions (Prisma)

```prisma
model ScheduledProject {
  id                    String         @id @default(uuid())
  name                  String
  description           String?
  deliveryDate          DateTime
  scheduledStartDate    DateTime?
  status                ProjectStatus  @default(draft)
  priority              TaskPriority   @default(NORMAL)
  tags                  String[]
  estimatedEffortHours  Float?
  actualEffortHours     Float          @default(0)
  createdBy             String
  createdAt             DateTime       @default(now())
  updatedAt             DateTime       @updatedAt
  completedAt           DateTime?

  phases                ProjectPhaseRecord[]

  @@index([status])
  @@index([priority])
  @@index([deliveryDate])
  @@index([createdBy])
  @@map("scheduled_projects")
}

model ProjectPhaseRecord {
  id                    String         @id @default(uuid())
  projectId             String
  phase                 SDLCPhase
  status                PhaseStatus    @default(pending)
  assignedAgentId       String?
  startedAt             DateTime?
  completedAt           DateTime?
  outputArtifacts       Json           @default("[]")
  notes                 String?
  durationMinutes       Int?

  project               ScheduledProject @relation(fields: [projectId], references: [id], onDelete: Cascade)

  @@unique([projectId, phase])
  @@index([status])
  @@index([assignedAgentId])
  @@map("project_phases")
}

enum SDLCPhase {
  requirements
  architecture
  development
  security
  testing
  deployment
  acceptance
}

enum PhaseStatus {
  pending
  in_progress
  completed
  failed
  blocked
  skipped
}

enum ProjectStatus {
  draft
  scheduled
  in_progress
  blocked
  completed
  failed
  cancelled
}
```

## 7. Relationship to v1 Foundation

The v2 layer builds on v1 without modifying it:

- **ScheduledProject** uses **ScheduledWorkItem** internally for each phase execution
- **ProjectOrchestrationService** calls **SchedulingService** to queue phase work
- **AgentPoolService** wraps the existing **AgentFactory** with pool semantics
- **Agent memory** from v1 is reused for cross-project learning
- **Triggers** from v1 can create ScheduledProjects (not just work items)
