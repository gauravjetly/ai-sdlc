## 1. Executive Summary

This architecture defines a **Scheduled Agent Work Builder** system that extends the existing AI-SDLC platform with UI-driven work scheduling, trigger-based multi-app creation, and memory-persistent agents. The design follows the existing layered architecture pattern and extends the current `Scheduler`, `EventManager`, and `BaseAgent` infrastructure with minimal breaking changes.

## 3. Key Architectural Decisions

### Decision 1: Extend Existing Infrastructure vs. New Microservice
**Decision**: Extend the existing monolithic platform services (not a new microservice).
**Rationale**: The platform is already structured with clear separation. A new module within the existing application avoids operational complexity. See ADR-022.

### Decision 2: BullMQ for Reliable Scheduling + node-cron for Cron Evaluation
**Decision**: Use BullMQ (backed by Redis) for reliable scheduled job execution; keep node-cron for cron expression parsing and schedule calculation.
**Rationale**: BullMQ provides reliable queue semantics with retry, delay, and persistence. node-cron is already in use and excellent for cron parsing. See ADR-023.

### Decision 3: PostgreSQL JSONB for Agent Memory
**Decision**: Store agent memory in PostgreSQL using JSONB columns within the existing database.
**Rationale**: Avoids introducing a new data store. JSONB provides flexible schema with indexing. Memory size limits (10GB/agent) are manageable. See ADR-024.

### Decision 4: Event-Driven Trigger Architecture
**Decision**: Extend the existing EventManager with trigger evaluation capability.
**Rationale**: The EventManager already handles platform events. Adding trigger evaluation is a natural extension. See ADR-025.

## 4. Detailed Component Design

### 4.1 Scheduling Engine

```
+---------------------------------------------+
|           SchedulingService                  |
+---------------------------------------------+
|                                             |
|  createScheduledWork(dto) -> WorkItem       |
|  updateScheduledWork(id, dto) -> WorkItem   |
|  pauseWork(id) -> void                      |
|  resumeWork(id) -> void                     |
|  cancelWork(id) -> void                     |
|  getScheduledWork(filters) -> WorkItem[]    |
|  getDashboard(userId) -> DashboardData      |
|                                             |
+---------------------+-----------------------+
                      |
          +-----------+-----------+
          |                       |
+---------v---------+   +---------v---------+
| CronSchedulerAdapter|   | BullMQSchedulingQ |
| (node-cron)        |   | (Redis-backed)    |
+-------------------+   +-------------------+
| calculateNextRun() |   | enqueueWork()     |
| validateCron()     |   | processWork()     |
| parseCron()        |   | retryWork()       |
+-------------------+   | delayWork()       |
                         +-------------------+
```

**Flow for Recurring Schedule**:
1. User creates work item with cron expression via UI
2. SchedulingService validates cron, stores in DB via Prisma
3. CronSchedulerAdapter calculates next run time
4. BullMQSchedulingQueue creates a delayed job for that time
5. When due, SchedulingWorker picks up the job
6. Worker invokes the appropriate agent via AgentFactory
7. After completion, CronSchedulerAdapter calculates next run
8. Loop continues until schedule is paused or deleted

**Flow for One-time Schedule**:
1. User sets specific datetime
2. SchedulingService stores in DB
3. BullMQSchedulingQueue creates a delayed job for that time
4. Worker executes at scheduled time
5. Status marked as completed

### 4.2 Trigger Engine

```
+---------------------------------------------+
|           TriggerService                     |
+---------------------------------------------+
|                                             |
|  createTrigger(dto) -> WorkTrigger          |
|  updateTrigger(id, dto) -> WorkTrigger      |
|  enableTrigger(id) -> void                  |
|  disableTrigger(id) -> void                 |
|  evaluateTriggers(event) -> WorkItem[]      |
|                                             |
+---------------------+-----------------------+
                      |
          +-----------+-----------+
          |                       |
+---------v---------+   +---------v---------+
| TriggerEvaluator  |   | EventManager      |
|                   |   | (existing)        |
+-------------------+   +-------------------+
| match(event,cond) |   | subscribe()       |
| evaluate(trigger) |   | publish()         |
| rateLimit()       |   | registerHandler() |
+-------------------+   +-------------------+
```

**Trigger Evaluation Flow**:
1. Platform event occurs (e.g., deployment.completed)
2. EventManager publishes event to registered handlers
3. TriggerEvaluator receives the event
4. For each active trigger matching the event type:
   a. Evaluate conditions against event data
   b. Apply rate limiting (max 10 fires per minute per trigger)
   c. If conditions match, create work items per trigger actions
5. Work items are enqueued via BullMQ for immediate or delayed execution
6. Trigger fire count and last-fired timestamp updated

**Condition Evaluation**:
```typescript
interface TriggerCondition {
  field: string;           // e.g., "event.data.environment"
  operator: 'eq' | 'neq' | 'contains' | 'gt' | 'lt' | 'in' | 'regex';
  value: string | number | string[];
}

// Conditions are ANDed together
// Multiple condition groups can be ORed
interface TriggerConditionGroup {
  operator: 'AND' | 'OR';
  conditions: TriggerCondition[];
}
```

### 4.3 Agent Memory System

```
+---------------------------------------------+
|           AgentMemoryService                 |
+---------------------------------------------+
|                                             |
|  store(agentId, key, value, ttl?) -> void   |
|  retrieve(agentId, key) -> MemoryEntry      |
|  retrieveAll(agentId) -> MemoryEntry[]      |
|  search(agentId, query) -> MemoryEntry[]    |
|  delete(agentId, key) -> void               |
|  getUsage(agentId) -> MemoryStats           |
|  evictExpired() -> void                     |
|                                             |
+---------------------+-----------------------+
                      |
          +-----------+-----------+
          |                       |
+---------v---------+   +---------v---------+
| PrismaAgentMemory |   | MemoryCache       |
| Repository        |   | (In-Process LRU)  |
+-------------------+   +-------------------+
| PostgreSQL JSONB  |   | Hot memory cache  |
| Full persistence  |   | 100 items/agent   |
| GIN index         |   | 5-min TTL         |
+-------------------+   +-------------------+
```

**Memory Architecture**:
- **L1 Cache**: In-process LRU cache (100 items per agent, 5-minute TTL)
- **L2 Storage**: PostgreSQL JSONB with GIN indexes for fast JSON queries
- **Eviction**: Expired entries cleaned up every hour via scheduled job
- **Namespace**: Memory keys are namespaced per agent: `{agentId}:{contextKey}`
- **Size Limit**: 10GB per agent (enforced via periodic size check)

**Memory Integration with BaseAgent**:
```typescript

  protected async recall(key: string): Promise<any | null>;
```

### 4.4 Multi-App Template System

```
+---------------------------------------------+
|           TemplateService                    |
+---------------------------------------------+
|                                             |
|  createTemplate(dto) -> WorkTemplate        |
|  instantiateTemplate(id, params) -> Work[]  |
|  listTemplates(filters) -> WorkTemplate[]   |
|  getTemplate(id) -> WorkTemplate            |
|                                             |
+---------------------------------------------+

WorkTemplate {
  id: UUID
  name: string
  description: string
  apps: AppDefinition[]  // Array of app configs
  dependencies: Dependency[]  // Inter-app dependencies
  variables: TemplateVariable[]  // Parameterizable values
}

AppDefinition {
  name: string
  agentType: AgentType
  taskType: string
  parameters: Record<string, any>
  dependsOn: string[]  // Names of other apps this depends on
}
```

**Multi-App Execution Strategy**:
1. Template is instantiated with user-provided parameters
2. Dependency graph is resolved (topological sort)
3. Independent apps are started in parallel
4. Dependent apps wait for their dependencies to complete
5. If an app fails, dependent apps are held
6. User can retry failed apps or rollback completed ones

## 8. Security Architecture

### Authentication and Authorization
- All API endpoints require valid JWT token (existing auth middleware)
- Work items are scoped to the creating user
- Admin role can view/manage all work items
- Agent memory is isolated per agent (no cross-agent access without explicit sharing)

### Input Validation
- Cron expressions validated via `node-cron.validate()` before storage
- Trigger conditions validated against a whitelist of allowed operators
- JSON payloads validated via Zod schemas
- SQL injection prevented by Prisma parameterized queries

### Trigger Security
- Rate limiting: max 10 trigger fires per minute per trigger
- Circuit breaker: disable trigger after 50 consecutive failures
- Condition evaluation uses safe JSON path traversal (no eval)
- Maximum 20 work items per single trigger fire

### Agent Memory Security
- Memory entries encrypted at rest (PostgreSQL TDE or application-level AES-256)
- Size limits enforced per agent (10GB default)
- TTL support for automatic expiration
- No cross-agent memory access unless explicitly configured

## 11. Error Handling and Recovery

### Execution Failure Recovery
1. BullMQ automatic retry with exponential backoff (3 retries, 30s/60s/120s)
2. After max retries, mark as failed and notify user
3. Failed work items can be manually retried from the dashboard

### Missed Schedule Recovery
1. On system startup, check for missed schedules
2. Execute any that should have run during downtime (configurable: skip or catch-up)
3. Log missed executions for audit

### Trigger Storm Protection
1. Rate limit per trigger (configurable, default 10/min)
2. Global rate limit (100 trigger fires per minute total)
3. Circuit breaker on cascading trigger chains (max depth 3)
