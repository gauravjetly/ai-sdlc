# Architecture: Phase 4 - Local Production Package

**ID**: ARCH-20260217-1507
**Date**: 2026-02-17
**Status**: APPROVED
**Author**: Architect Agent (Jets)

## 1. Executive Summary

Phase 4 transforms the existing AI-SDLC platform from a development-time codebase
into a production-grade, locally-installable package. The key architectural changes are:

1. **SQLite provider** added alongside existing file/postgres providers (zero-config local DB)
2. **In-memory LRU cache** added alongside existing file/bullmq event bus providers
3. **Unified CLI** wrapping all platform commands into a single `aisdlc` binary
4. **npm monorepo** packaging with workspaces for modular distribution
5. **Health check system** providing production-grade self-diagnostics
6. **Cross-platform installer** for Mac/Linux/Windows

## 2. System Architecture

### 2.1 Package Structure (npm Workspaces Monorepo)

```
packages/
  @aisdlc/
    core/               # Shared types, utilities, config management
    agents/             # 12 agent definition loader + runtime
    agent-mesh/         # Wraps src/agent-mesh with SQLite provider
    integration/        # Wraps src/integration (classifier, router, governance)
    cli/                # Commander-based CLI tool
    dashboard/          # Vite-built dashboard static assets + server
    platform/           # Main entry point - orchestrates all packages
```

### 2.2 Provider Architecture (Extended)

The existing provider pattern (file | postgres for storage, file | bullmq for event bus)
is extended with new lightweight providers:

```
StorageProvider: 'file' | 'sqlite' | 'postgres'
EventBusProvider: 'file' | 'memory' | 'bullmq'
```

Default for local production: `STORAGE=sqlite`, `EVENT_BUS=file`

### 2.3 SQLite Provider Design

**Database Location**: `~/.aisdlc/data/platform.db`

The SQLite provider implements the same `MemoryService` interface as the
PostgresMemoryProvider, ensuring zero code changes in the AgentMesh class.

**Key Design Decisions**:
- Use `better-sqlite3` (synchronous, embedded, no native compilation issues on most platforms)
- WAL mode enabled for concurrent read access
- JSON columns stored as TEXT with JSON functions for querying
- No vector extension - use TF-IDF keyword scoring for local search instead of pgvector
- Migrations run automatically on first connection

### 2.4 In-Memory Cache Design

The in-memory cache replaces Redis for local production. It implements
a simple LRU eviction strategy with TTL support.

**Interface**: Matches the EventBus interface from `src/agent-mesh/bus/event-bus.ts`

**Implementation**: Standard Map-based LRU with:
- Max 10,000 entries (configurable)
- TTL per entry (default 1 hour)
- Automatic cleanup via setInterval every 60 seconds
- getStats() for monitoring

This is NOT a new event bus provider. The existing file-based event bus
provider is sufficient for local production. The in-memory cache is used
for the ClassificationCache in the integration layer.

```
aisdlc (Commander.js)
  |-- init           # Initialize ~/.aisdlc/ directory, run migrations, write config
  |-- start          # Start dashboard server (express static), open browser
  |-- stop           # Stop dashboard server (PID file)
  |-- status         # Show platform health, agent count, DB stats
  |-- dashboard      # Open dashboard URL in default browser
  |-- doctor         # Run comprehensive health checks (8 checks)
  |-- config         # Show/edit platform configuration
  |-- hooks:install  # Copy hook files to ~/.claude/hooks/
  |-- hooks:remove   # Remove hook files
  |-- mcp:configure  # Write MCP server config to claude_desktop_config.json
  |-- logs           # Tail audit log from SQLite
  |-- reset          # Reset database and config to defaults
  |-- uninstall      # Remove ~/.aisdlc/ and hooks
  |-- version        # Show version
  |-- workflow        # Execute SDLC workflow commands
```

### 2.6 Health Check System

Eight health checks run in parallel:

| Check | What it validates |
|-------|-------------------|
| node-version | Node.js >= 20 |
| claude-code | `claude` CLI available |
| database | SQLite DB accessible and migrations current |
| hooks | Hook files exist in ~/.claude/hooks/ |
| mcp | MCP server config present |
| agents | All 12 agent definitions loadable |
| dashboard | Dashboard build assets present |
| config | Platform config valid |

### 2.7 Dashboard Serving

The dashboard (already built in Phase 3 as a Vite/React app) is served locally
by a lightweight Express static server on port 3030. The CLI `aisdlc start`
launches this server as a background process (PID stored in `~/.aisdlc/dashboard.pid`).

## 3. Architecture Decision Records

### ADR-001: SQLite over PostgreSQL for Local Production

**Context**: The platform needs a database that works without installation.
**Decision**: Add SQLite as a third storage provider.
**Rationale**: SQLite is file-based, requires no server process, works on all platforms,
and handles concurrent reads via WAL mode. For single-developer local use,
it provides more than sufficient performance.
**Consequences**: No vector search capability locally (TF-IDF keyword search instead).
Postgres remains available for team/cloud deployments.

### ADR-002: npm Workspaces Monorepo

**Context**: The platform has multiple logical packages that need independent versioning.
**Decision**: Use npm workspaces (built into npm 7+) rather than Lerna/Nx/Turborepo.
**Rationale**: Zero additional tooling required. npm workspaces are mature and sufficient
for our package count (~7 packages). Developers already have npm installed.
**Consequences**: Simple `npm install` at root installs all workspace dependencies.

### ADR-003: better-sqlite3 over sql.js

**Context**: Need an embedded SQLite library for Node.js.
**Decision**: Use `better-sqlite3` as primary, with `sql.js` as WASM fallback.
**Rationale**: `better-sqlite3` is synchronous (no callback complexity), faster than
sql.js, and prebuilt binaries exist for Mac/Linux/Windows. However, if native
compilation fails, the system falls back to sql.js (pure WASM).
**Consequences**: Installation may require node-gyp on some systems. The fallback
mitigates this risk.

### ADR-004: Express Static Server for Dashboard

**Context**: Dashboard needs to be served locally.
**Decision**: Use a minimal Express server to serve the pre-built Vite assets.
**Rationale**: Express is already a dependency. A 20-line static file server is
simpler than requiring developers to run Vite dev server. The server also
provides the WebSocket endpoint for real-time updates.
**Consequences**: Dashboard runs on localhost:3030. No HMR in production mode.

### ADR-005: File-Based Event Bus as Default

**Context**: Need an event bus that works without Redis.
**Decision**: Use existing file-based event bus provider as default for local production.
**Rationale**: The file-based provider already exists, is tested, and works without
any external dependencies. For single-developer use, file I/O performance is adequate.
**Consequences**: No need for a new in-memory event bus provider. The in-memory cache
is used only for the classification cache (separate concern).

## 4. Data Flow

```
Developer types in Claude Code
  |
  v

  |
  v
HookBridge -> Request Classifier (rule-based + optional LLM)
  |
  v

  |
  v

  |                      |
  v                      v
SQLite (audit, knowledge)  File EventBus (events)
  |
  v

```

## 5. Cross-Platform Considerations

| Concern | Solution |
|---------|----------|
| Path separators | Use `path.join()` everywhere, never hardcode `/` |
| Home directory | Use `os.homedir()`, not `$HOME` |
| Process management | PID files for dashboard, `cross-spawn` for child processes |
| File permissions | Skip chmod on Windows |
| SQLite native | Prebuild binaries + WASM fallback |
| Shell scripts | Provide both .sh and .ps1, CLI handles abstraction |

## 6. Security Considerations

- SQLite database file permissions: 0600 (owner only)
- No credentials stored in database (API keys stay in env vars)
- CLI does not accept untrusted input for SQL queries (parameterized only)
- Dashboard binds to localhost only (not 0.0.0.0)
- Bypass tokens use HMAC-SHA256 (existing Phase 3 implementation)

## 7. Migration Path

Existing users upgrading from Phases 1-3:
1. `aisdlc init` detects existing `~/.claude/` installation
2. Migrates file-based data to SQLite if user consents
3. Preserves all existing agent definitions and configuration
4. No breaking changes to existing hooks or MCP config

## 8. Performance Targets

| Metric | Target |
|--------|--------|
| Installation time | < 2 minutes |
| `aisdlc start` (cold) | < 3 seconds |
| Classification (cached) | < 5ms |
| Classification (uncached) | < 200ms |
| Dashboard load | < 1 second |
| SQLite query (typical) | < 10ms |
| Health check (all 8) | < 5 seconds |
| Memory footprint (idle) | < 50MB |

- `open` - Cross-platform browser opener

- `ora` - Terminal spinners

Removed/optional dependencies:
- `pg`, `pg-pool` - Optional (only for postgres mode)
- `bullmq`, `ioredis` - Optional (only for bullmq mode)
- `pgvector` - Optional (only for postgres mode)
