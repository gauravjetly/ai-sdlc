## 1. Executive Summary

This document defines the component architecture for the Network Architect Wizard in the Infrastructure Designer module. The wizard provides a guided, step-by-step interface for designing AWS network infrastructure including VPCs, subnets, routing, and security groups.

### Design Principles
1. **Component Isolation**: Each wizard step is an independent, testable component
2. **State Centralization**: All state flows through DesignWizardContext
3. **Validation at Boundaries**: Validate on field blur and step transition
4. **Accessibility First**: WCAG 2.1 AA compliance in all components
5. **Performance Optimization**: Memoization and lazy loading where beneficial

## 3. Component Specifications

### 3.1 WizardDrawer

**Purpose**: Main drawer container that wraps step content

**Props**:

**Responsibilities**:

**Responsibilities**:

### 3.3 VPCConfigurationStep

**Purpose**: Configure VPC settings

**Props**:
```typescript
interface VPCConfigurationStepProps {
  vpc: VPCConfig;
  onChange: (vpc: VPCConfig) => void;
  errors: ValidationError[];
```

**Key Features**:
- VPC name input with validation (alphanumeric, hyphens, 3-64 chars)
- CIDR selector with preset options + custom input
- DNS resolution and hostnames toggles
- IPv6 toggle (optional)
- Tags editor
- Available IPs calculator display

**Validation Rules**:
- Name: Required, alphanumeric with hyphens
- CIDR: Required, valid RFC 1918 range, /16 to /28

**Purpose**: Design subnet topology

**Props**:
```typescript

  vpcCidr: string;
  region: string;
  subnets: SubnetConfig[];
  onChange: (subnets: SubnetConfig[]) => void;
  errors: ValidationError[];
```

**Key Features**:
- Subnet list with visual indicators (public/private, AZ)
- Add subnet form with auto-CIDR suggestions
- Edit/delete existing subnets
- Multi-AZ distribution indicator
- CIDR overlap detection

**Validation Rules**:
- At least 1 subnet required
- Subnet CIDR must be within VPC CIDR
- No overlapping subnet CIDRs
- Valid AZ selection for region

### 3.5 RoutingConfigurationStep

**Purpose**: Configure gateways and route tables

**Props**:
```typescript
interface RoutingConfigurationStepProps {
  subnets: SubnetConfig[];
  routing: RoutingConfig;
  onChange: (routing: RoutingConfig) => void;
  errors: ValidationError[];
}
```

**Key Features**:
- Internet Gateway toggle and name
- NAT Gateway configuration with subnet selection
- Route table list with routes
- Subnet-to-route table associations
- Cost estimate for NAT Gateways

**Auto-Configuration Logic**:
- If public subnets exist, suggest IGW
- If private subnets exist, suggest NAT
- Auto-create public and private route tables
- Auto-add local route to both tables
- Auto-add IGW route to public table
- Auto-add NAT route to private table

### 3.6 SecurityGroupsStep

**Purpose**: Configure security group rules

**Props**:
```typescript
interface SecurityGroupsStepProps {
  securityGroups: SecurityGroupConfig[];
  onChange: (sgs: SecurityGroupConfig[]) => void;
  errors: ValidationError[];
}
```

**Key Features**:
- Security group list with rule counts
- Add/edit security group form
- Ingress rules table with add/edit/delete
- Egress rules table with add/edit/delete
- Common rule templates (HTTP, HTTPS, SSH, etc.)
- Security warnings for risky rules

**Validation Rules**:
- At least 1 security group required
- No unrestricted SSH (22) or RDP (3389) to 0.0.0.0/0
- Valid CIDR or security group reference
- Valid port range (1-65535)

### 3.7 NetworkValidationStep

**Purpose**: Validate and review complete network configuration

**Props**:
```typescript
interface NetworkValidationStepProps {
  vpc: VPCConfig;
  subnets: SubnetConfig[];
  routing: RoutingConfig;
  securityGroups: SecurityGroupConfig[];
  onValidationComplete: (result: ValidationResult) => void;
}
```

**Key Features**:
- Automatic validation on mount
- Grouped display of errors, warnings, info
- "Fix" buttons that navigate to relevant step
- Configuration summary (JSON/YAML)
- Network topology diagram preview
- Terraform code preview modal
- Deploy button (enabled only when valid)

**Validation Categories**:
1. **Errors** (must fix):
   - CIDR overlap
   - Missing route table associations
   - Unrestricted sensitive ports
   - Invalid configurations

2. **Warnings** (should fix):
   - Single AZ deployment
   - Missing NAT for private subnets
   - Overly permissive egress rules

3. **Info** (best practices):
   - Consider VPC Flow Logs
   - Consider multiple NAT Gateways for HA
   - Consider tagging resources

## 6. ReactFlow Integration

### 6.1 Node Generation

When network layer is complete, generate nodes for ReactFlow:

```typescript
function generateNetworkNodes(config: NetworkLayerData): DesignNode[] {
  const nodes: DesignNode[] = [];

  // VPC container node
  nodes.push({
    id: `vpc-${config.vpc.name}`,
    type: 'vpcNode',
    position: { x: 50, y: 50 },
    data: { ...config.vpc },
    layer: 'network',
    width: 800,
    height: 600,
  });

  // Group subnets by AZ
  const azGroups = groupBy(config.subnets, 'availabilityZone');
  let azX = 100;

  Object.entries(azGroups).forEach(([az, subnets]) => {
    let subnetY = 100;
    subnets.forEach((subnet) => {
      nodes.push({
        id: `subnet-${subnet.id}`,
        type: 'subnetNode',
        position: { x: azX, y: subnetY },
        data: { ...subnet, az },
        layer: 'network',
        parentId: `vpc-${config.vpc.name}`,
      });
      subnetY += 120;
    });
    azX += 200;
  });

  // Internet Gateway
  if (config.routing.internetGateway.enabled) {
    nodes.push({
      id: 'igw',
      type: 'igwNode',
      position: { x: 400, y: -50 },
      data: { name: config.routing.internetGateway.name },
      layer: 'network',
    });
  }

  // NAT Gateways
  config.routing.natGateways.forEach((nat, i) => {
    nodes.push({
      id: `nat-${nat.id}`,
      type: 'natNode',
      position: { x: 100 + i * 200, y: 50 },
      data: { ...nat },
      layer: 'network',
    });
  });

  return nodes;
}
```

## 11. ADRs (Architecture Decision Records)

### ADR-025: Network Wizard Step Component Pattern

**Status**: Accepted

**Context**: Need a consistent pattern for wizard step components

**Decision**: Each step receives props for its data slice and an onChange callback. Steps manage local form state and commit to parent on submit.

**Consequences**:
- Consistent API across all steps
- Easy testing of individual steps
- Clear data flow

### ADR-026: CIDR Validation Library

**Status**: Accepted

**Context**: Need reliable CIDR parsing and validation

**Decision**: Implement custom CIDR utilities using bitwise operations rather than external library

**Consequences**:
- No external dependencies
- Full control over behavior
- Requires thorough testing

### ADR-027: Validation Timing

**Status**: Accepted

**Context**: When to run validation (on every keystroke vs blur vs submit)

**Decision**: Validate on blur for fields, on submit for forms, on "Next" for steps

**Consequences**:
- Good UX (not intrusive)
- Performance efficient
- Clear error timing
