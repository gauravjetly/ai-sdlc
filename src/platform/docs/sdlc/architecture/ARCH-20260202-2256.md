## 1. Executive Summary

This document defines the architecture for the Environment Management UI components, enabling infrastructure teams to manage multi-environment configurations (dev/staging/prod) within the Infrastructure Designer module.

### 1.1 Design Principles

| Principle | Application |
|-----------|-------------|
| **Single Source of Truth** | DesignWizardContext as central state |
| **Configuration Inheritance** | Prod extends staging extends dev |
| **Least Privilege** | Production requires explicit authorization |
| **Audit Everything** | All changes logged with user context |
| **Optimistic Updates** | UI updates immediately, reconciles with backend |

// useEnvironmentConfig.ts
export function useEnvironmentConfig(environment: Environment) {
  const { environments, selectedEnvironment, updateEnvironmentConfig } = useDesignWizard();

  const config = useMemo(() => environments[environment], [environments, environment]);

  const effectiveConfig = useMemo(() => {
    // Merge base config with overrides
    return mergeConfigurations(environments.dev, config.overrides);
  }, [config]);

  const setOverride = useCallback((path: string, value: any) => {
    updateEnvironmentConfig(environment, {
      overrides: {
        ...config.overrides,
        [path]: {
          originalValue: getValueAtPath(environments.dev, path),
          overrideValue: value,
          modifiedBy: currentUser.id,
          modifiedAt: new Date(),
        },
      },
    });
  }, [environment, config, updateEnvironmentConfig]);

  return { config, effectiveConfig, setOverride };
}

## 5. Data Flow

### 5.1 Configuration Inheritance Model

```
┌──────────────────┐
│   Base Config    │  <- Defined in network/platform/devops layers
│   (from dev)     │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ Staging Overrides│  <- Environment-specific modifications
│  instanceType    │
│  replicas        │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ Effective Config │  <- merge(base, overrides)
│   (computed)     │
└──────────────────┘
```

### 5.2 Configuration Merge Algorithm

```typescript
function mergeConfiguration(
  base: EnvironmentConfig,
  overrides: ConfigOverrides
): EffectiveConfig {
  const effective = deepClone(base);

  for (const [path, override] of Object.entries(overrides)) {
    setValueAtPath(effective, path, override.overrideValue);
  }

  return effective;
}

function calculateCostDelta(
  base: EnvironmentConfig,
  effective: EffectiveConfig
): CostDelta {
  const baseCost = estimateCost(base);
  const effectiveCost = estimateCost(effective);

  return {
    baseCost,
    effectiveCost,
    delta: effectiveCost - baseCost,
    percentChange: ((effectiveCost - baseCost) / baseCost) * 100,
  };
}
```

### 8.2 Error Recovery

```typescript
// Optimistic update with rollback
async function updateConfigWithRollback(
  environment: Environment,
  changes: ConfigOverrides
) {
  const previousState = getSnapshot(environment);

  // Optimistic update
  dispatch({ type: 'UPDATE_ENV_CONFIG', payload: { env: environment, config: changes } });

  try {
    await api.updateEnvironmentConfig(environment, changes);
  } catch (error) {
    // Rollback on failure
    dispatch({ type: 'UPDATE_ENV_CONFIG', payload: { env: environment, config: previousState } });

    notifyError({
      title: 'Configuration Update Failed',
      message: error.message,
      action: {
        label: 'Retry',
        onClick: () => updateConfigWithRollback(environment, changes),
      },
    });
  }
}
```

## 10. Architecture Decision Records

### ADR-ENV-001: Configuration Inheritance Strategy

**Context**: Need to manage environment configurations efficiently.

**Decision**: Use base-plus-overrides model where dev is the base and other environments extend it.

**Rationale**:
- Reduces configuration drift
- Clear audit trail of what's different
- Easy to see effective configuration

**Consequences**:
- Deleting a dev config affects all environments
- Need careful migration when changing base

### ADR-ENV-002: Secret Management Approach

**Context**: Need to manage sensitive configuration values.

**Decision**: Store secret references (ARNs) only in UI; actual values stored in AWS Secrets Manager.

**Rationale**:
- Secrets never transmitted to frontend
- Leverage AWS's security infrastructure
- Support secret rotation

**Consequences**:
- AWS dependency for secrets
- Additional latency for secret retrieval at deployment time

### ADR-ENV-003: Promotion Approval Model

**Context**: Production changes need governance.

**Decision**: Require explicit approval for production promotions; auto-approve for non-production.

**Rationale**:
- Prevents accidental production changes
- Maintains audit trail
- Supports compliance requirements

**Consequences**:
- Slower production deployment
- Need to define approvers
