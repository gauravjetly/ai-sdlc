## 1. Executive Summary

This document defines the architecture for 7 backend services that power the Infrastructure Designer module. The architecture follows Clean Architecture principles with clear separation of concerns, dependency injection, and robust error handling.

### 1.1 Architecture Goals

1. **Modularity**: Each service is independently deployable and testable
2. **Scalability**: Support 100+ concurrent users with sub-500ms response times
3. **Reliability**: 99.9% uptime with graceful degradation
4. **Maintainability**: Clean code with >90% test coverage
5. **Security**: Defense in depth with audit logging

### 1.2 Key Architectural Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture Style | Layered (Clean Architecture) | Clear separation, testability |
| Database Access | Prisma ORM | Type safety, migrations |
| Async Processing | Bull Queue | Reliable job processing |
| Real-time Updates | WebSocket | Low latency notifications |
| IaC Execution | Terraform CLI | Industry standard, AWS support |

### 3.1 WorkflowManager Service

**Purpose**: Orchestrate multi-step wizard workflow for infrastructure design.

**Architecture Pattern**: State Machine

```typescript

enum WorkflowState {
  INITIALIZED,
  NETWORK_IN_PROGRESS,
  NETWORK_COMPLETE,
  PLATFORM_IN_PROGRESS,
  PLATFORM_COMPLETE,
  DEVOPS_IN_PROGRESS,
  DEVOPS_COMPLETE,
  READY_TO_DEPLOY
}

// State Transitions
INITIALIZED -> NETWORK_IN_PROGRESS (startNetwork)
NETWORK_IN_PROGRESS -> NETWORK_COMPLETE (completeNetwork)
NETWORK_COMPLETE -> PLATFORM_IN_PROGRESS (startPlatform)
PLATFORM_IN_PROGRESS -> PLATFORM_COMPLETE (completePlatform)
PLATFORM_COMPLETE -> DEVOPS_IN_PROGRESS (startDevOps)
DEVOPS_IN_PROGRESS -> DEVOPS_COMPLETE (completeDevOps)
DEVOPS_COMPLETE -> READY_TO_DEPLOY (all layers complete)

*_COMPLETE -> *_IN_PROGRESS (rollbackLayer)
```

**Dependency Resolution (Topological Sort)**:

```typescript
// Layer dependencies (DAG)
const LAYER_DEPS = {
  network: [],
  platform: ['network'],
  devops: ['platform']
};

function topologicalSort(layers: LayerType[]): LayerType[] {
  const sorted: LayerType[] = [];
  const visited = new Set<LayerType>();
  const visiting = new Set<LayerType>();

  function visit(layer: LayerType) {
    if (visiting.has(layer)) {
      throw new Error('Circular dependency detected');
    }
    if (visited.has(layer)) return;

    visiting.add(layer);
    for (const dep of LAYER_DEPS[layer]) {
      if (layers.includes(dep)) {
        visit(dep);
      }
    }
    visiting.delete(layer);
    visited.add(layer);
    sorted.push(layer);
  }

  for (const layer of layers) {
    visit(layer);
  }

  return sorted;
}
```

**Cost Calculation Strategy**:

```typescript
// Strategy interface
interface ResourceCostCalculator {
  resourceType: string;
  calculate(config: ResourceConfig, region: string): Promise<MonthlyCost>;
}

// Concrete calculators
class EC2CostCalculator implements ResourceCostCalculator {
  async calculate(config, region) {
    const basePrice = await this.getInstancePrice(config.instanceType, region);
    const hours = 730; // avg hours per month
    const storageCost = config.storageGB * this.getEBSPrice(config.storageType);
    return {
      compute: basePrice * hours,
      storage: storageCost,
      total: basePrice * hours + storageCost
    };
  }
}

class RDSCostCalculator implements ResourceCostCalculator {
  async calculate(config, region) {
    const instanceCost = await this.getDBInstancePrice(config);
    const storageCost = config.storageGB * this.getStoragePrice(config);
    const ioCost = config.estimatedIOPS * this.getIOPrice();
    const multiAZMultiplier = config.multiAZ ? 2 : 1;
    return {
      instance: instanceCost * 730 * multiAZMultiplier,
      storage: storageCost,
      io: ioCost,
      total: (instanceCost * 730 * multiAZMultiplier) + storageCost + ioCost
    };
  }
}
```

## 11. ADRs (Architecture Decision Records)

### ADR-001: Use Prisma for Database Access
- **Context**: Need type-safe database access with migrations
- **Decision**: Use Prisma ORM
- **Consequences**: Strong typing, automatic migrations, some query limitations

### ADR-002: Bull Queue for Async Processing
- **Context**: Need reliable async job processing for deployments
- **Decision**: Use Bull with Redis
- **Consequences**: Redis dependency, excellent job management, built-in retry

### ADR-003: Terraform CLI for Infrastructure
- **Context**: Need to deploy AWS infrastructure
- **Decision**: Execute Terraform CLI via child process
- **Consequences**: Requires Terraform installed, well-documented, mature ecosystem

### ADR-004: Event-Driven State Management
- **Context**: Need real-time updates for UI
- **Decision**: Use EventEmitter with WebSocket bridge
- **Consequences**: Loose coupling, easy testing, requires WebSocket infrastructure

### ADR-005: Strategy Pattern for Terraform Generators
- **Context**: Need to support 27+ node types
- **Decision**: Use Strategy pattern with one generator per node type
- **Consequences**: Easy to add new types, clear separation, some code duplication
