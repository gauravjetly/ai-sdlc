## 1. Executive Summary

This document defines the architecture for the DevOps Wizard components, which orchestrate CI/CD pipeline configuration, monitoring setup, service connections, Infrastructure as Code generation, and comprehensive validation for the Infrastructure Designer module.

### 1.1 Design Goals
1. **Consistency**: Follow established patterns from NetworkArchitectWizard
2. **Modularity**: Each step is an independent, testable component
3. **Extensibility**: Support future cloud providers and tools
4. **Performance**: Sub-second step transitions, efficient validation
5. **Accessibility**: WCAG 2.1 AA compliance throughout

### 1.2 Key Decisions
- 5-step wizard matching Network/Platform wizard structure
- Centralized state via DesignWizardContext
- Validation hook pattern for reusable validation logic
- Terraform generation as final deliverable
- Cross-layer validation for service connections

## 3. Component Design

### 3.1 DevOpsWizard.tsx (Orchestrator)

**Purpose**: Orchestrate the 5-step DevOps configuration workflow

**Props Interface**:
```typescript
interface DevOpsWizardProps {
  step: number;
  onStepChange: (step: number) => void;
  onComplete?: () => void;
  onSave?: () => Promise<void>;
}
```

**State Management**:
```typescript
// Local state (step-specific data)
const [cicd, setCICD] = useState<CICDConfig>(existingData?.cicd || DEFAULT_CICD_CONFIG);
const [monitoring, setMonitoring] = useState<MonitoringConfig>(existingData?.monitoring || DEFAULT_MONITORING_CONFIG);
const [connections, setConnections] = useState<ServiceConnectionConfig[]>(existingData?.connections || []);
const [iac, setIaC] = useState<IaCConfig>(existingData?.iac || DEFAULT_IAC_CONFIG);
const [validationResult, setValidationResult] = useState<DevOpsValidationResult | null>(null);

// Sync to context
useEffect(() => {
  updateLayerData('devops', {
    config: { cicd, monitoring, connections, iac, validation: validationResult },
    nodes: [],
    edges: [],
  });
}, [cicd, monitoring, connections, iac, validationResult]);
```

**Step Rendering**:
```typescript
const renderStep = () => {
  switch (step) {
    case 1: return <CICDPipelineStep cicd={cicd} onChange={setCICD} errors={stepErrors} />;
    case 2: return <MonitoringObservabilityStep monitoring={monitoring} onChange={setMonitoring} errors={stepErrors} />;
    case 3: return <ServiceConnectionsStep connections={connections} onChange={setConnections} networkData={networkLayerData} platformData={platformLayerData} errors={stepErrors} />;
    case 4: return <InfrastructureAsCodeStep iac={iac} onChange={setIaC} allLayerData={{ network: networkLayerData, platform: platformLayerData, devops: { cicd, monitoring, connections } }} errors={stepErrors} />;
    case 5: return <DevOpsValidationStep allData={allData} onFix={handleFix} onValidationComplete={handleValidationComplete} />;
    default: return <Alert severity="error">Invalid step</Alert>;
  }
};
```

## 5. Validation Hook Design

### 5.1 useDevOpsValidation.ts

```typescript
/**
 * DevOps Validation Hook
 * Provides comprehensive validation for DevOps layer configuration
 */

import { useCallback, useMemo } from 'react';
import {
  CICDConfig,
  MonitoringConfig,
  ServiceConnectionConfig,
  LoadBalancerConfig,
  IaCConfig,
  DevOpsLayerData,
  DevOpsValidationResult,
  ValidationError,
  CostValidationResult,
  ReadinessCheckItem,
} from '../../types/devops';
import { NetworkLayerData } from '../../types/network';
import { PlatformLayerData } from '../../types/platform';

export interface UseDevOpsValidationResult {
  // Full validation
  validate: (
    devopsData: DevOpsLayerData,
    networkData: NetworkLayerData,
    platformData: PlatformLayerData
  ) => DevOpsValidationResult;

  // Component-level validation
  validateCICDConfig: (config: CICDConfig) => ValidationError[];
  validateMonitoringConfig: (config: MonitoringConfig) => ValidationError[];
  validateConnections: (
    connections: ServiceConnectionConfig[],
    networkData: NetworkLayerData,
    platformData: PlatformLayerData
  ) => ValidationError[];
  validateLoadBalancer: (config: LoadBalancerConfig) => ValidationError[];
  validateIaCConfig: (config: IaCConfig) => ValidationError[];

  // Cross-layer validation
  validateServiceConnectivity: (
    connection: ServiceConnectionConfig,
    networkData: NetworkLayerData
  ) => ValidationError[];

  // Cost estimation
  estimateCosts: (
    devopsData: DevOpsLayerData,
    networkData: NetworkLayerData,
    platformData: PlatformLayerData
  ) => CostValidationResult;

  // Readiness check
  generateReadinessChecklist: (
    devopsData: DevOpsLayerData,
    networkData: NetworkLayerData,
```

## 8. Architecture Decision Records

### ADR-001: 5-Step Wizard Structure

**Context**: Need to design DevOps configuration workflow
**Decision**: Use 5-step wizard matching Network/Platform patterns
**Rationale**: Consistency, user familiarity, logical grouping
**Consequences**: Clear progression, may need to revisit if requirements expand

### ADR-002: Cross-Layer Validation

**Context**: DevOps layer must validate connections to Network/Platform
**Decision**: Validation hook accesses all layer data from context
**Rationale**: Single source of truth, real-time validation
**Consequences**: Requires complete layer data, may impact performance

### ADR-003: Terraform Generation Strategy

**Context**: Need to generate valid Terraform from wizard configuration
**Decision**: Generate modular Terraform with clear separation
**Rationale**: Maintainability, reusability, industry best practices
**Consequences**: More complex generation logic, but better output
