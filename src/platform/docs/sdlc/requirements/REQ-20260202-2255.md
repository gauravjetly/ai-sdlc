# Requirements: Layer Management UI Components

**Document ID**: REQ-20260202-2255
**Version**: 1.0
**Created**: 2026-02-02
**Status**: APPROVED

## 1. Executive Summary

### 1.1 Problem Statement
The Infrastructure Designer has completed implementation of three wizard layers (Network, Platform, DevOps) but lacks a unified interface for managing, visualizing dependencies between, and deploying these layers. Users need the ability to:
- Navigate between completed layers
- Visualize layer dependencies and deployment order
- Deploy layers with real-time feedback
- Manage layer configurations across environments
- Track deployment history and perform rollbacks

### 1.2 Solution Overview
Build five React components that provide a comprehensive layer management interface:
1. **LayerSelector** - Navigation and status overview
2. **LayerDeploymentPanel** - Deployment controls with real-time logs
3. **LayerDependencyGraph** - Visual dependency visualization
4. **LayerConfigPanel** - Configuration management and editing
5. **LayerTimeline** - Deployment history and audit trail

### 1.3 Success Criteria
- Users can deploy all three layers in correct dependency order
- Real-time deployment status updates via WebSocket
- Clear visualization of layer dependencies
- Ability to rollback failed deployments
- Environment-specific configuration management

## 3. Functional Requirements

### FR-001: Layer Selector Component
**Priority**: P0 (Critical)

#### FR-001.1: Layer Navigation
| ID | Requirement | Acceptance Criteria |
|----|-------------|---------------------|
| FR-001.1.1 | Display three layers (Network, Platform, DevOps) in visual hierarchy | Given the LayerSelector is rendered, When the user views the component, Then all three layers are displayed with their respective icons and colors |
| FR-001.1.2 | Show layer status badge (pending, complete, deployed, failed) | Given a layer has a status, When the selector is displayed, Then the correct status badge with appropriate color is shown |
| FR-001.1.3 | Allow clicking on a layer to select it | Given a layer is clickable, When the user clicks on a layer, Then that layer becomes the active/selected layer |
| FR-001.1.4 | Disable selection of layers with unmet dependencies | Given Platform layer requires Network completion, When Network is not complete, Then Platform layer click is disabled with tooltip explaining why |
| FR-001.1.5 | Show completion percentage for each layer | Given a layer has partial configuration, When viewing the selector, Then show percentage complete (e.g., "3/5 steps complete") |

#### FR-001.2: Visual Indicators
| ID | Requirement | Acceptance Criteria |
|----|-------------|---------------------|
| FR-001.2.1 | Use consistent layer colors (Network: blue, Platform: green, DevOps: orange) | Given the design system, When layers are displayed, Then colors match the LAYER_CONFIG in WizardDrawer |
| FR-001.2.2 | Show dependency arrows between layers | Given layers have dependencies, When the selector is rendered, Then visual arrows show Network -> Platform -> DevOps flow |
| FR-001.2.3 | Animate status transitions | Given a layer status changes, When the change occurs, Then smooth animation indicates the transition |

### FR-002: Layer Deployment Panel Component

#### FR-002.1: Deployment Controls
| ID | Requirement | Acceptance Criteria |

| FR-002.2.3 | Display estimated time remaining | Given deployment is in progress, When viewing status, Then estimated completion time is shown |

#### FR-002.4: Cost Display
| ID | Requirement | Acceptance Criteria |
|----|-------------|---------------------|

## 5. Data Model Integration

### 5.1 Existing Types (from DesignWizardContext)
```typescript
// Layer types from existing codebase
type LayerType = 'network' | 'platform' | 'devops' | 'fullstack';
type LayerStatus = 'pending' | 'complete' | 'deployed' | 'failed';
type Environment = 'dev' | 'staging' | 'prod';

// Layer state structure
interface LayerData {
  nodes: DesignNode[];
  edges: DesignEdge[];
  config?: Record<string, any>;
}
```

### 5.2 New Types Required
```typescript
// Deployment event for timeline
interface DeploymentEvent {
  id: string;
  layerType: LayerType;
  eventType: 'deploy' | 'rollback' | 'config_change' | 'validation';
  status: 'pending' | 'running' | 'success' | 'failed';
  startedAt: string;
  completedAt?: string;
  duration?: number;
  initiatedBy: string;
  environment: Environment;
  terraformPlan?: string;
  terraformOutput?: Record<string, any>;
  logs?: string[];
  error?: string;
}

// Layer dependency definition
interface LayerDependency {
  layer: LayerType;
  dependsOn: LayerType[];
  validationRules: ValidationRule[];
}

// Cost estimate structure
interface LayerCostEstimate {
  layerType: LayerType;
  environment: Environment;
  monthlyCost: number;
  breakdown: {
    resourceType: string;
    resourceName: string;
    monthlyCost: number;
  }[];
}
```

## 6. API Requirements

### 6.1 Existing Endpoints Used
- `GET /api/v1/workflows/:id` - Load workflow data
- `PUT /api/v1/workflows/:id` - Update workflow state
- `POST /api/v1/workflows/:id/deploy` - Deploy layer
- `POST /api/v1/workflows/:id/validate` - Validate layer
- `POST /api/v1/workflows/:id/advance` - Advance to next layer

### 6.2 New Endpoints Required
| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/v1/workflows/:id/rollback` | POST | Rollback layer deployment |
| `/api/v1/workflows/:id/timeline` | GET | Get deployment history |
| `/api/v1/workflows/:id/cost-estimate` | GET | Get cost estimates per layer |
| `/api/v1/workflows/:id/terraform-plan` | GET | Get Terraform plan preview |

### 6.3 WebSocket Events
| Event | Direction | Description |
|-------|-----------|-------------|
| `deployment:started` | Server->Client | Deployment initiated |
| `deployment:log` | Server->Client | Log line generated |
| `deployment:progress` | Server->Client | Progress percentage update |
| `deployment:completed` | Server->Client | Deployment finished |
| `deployment:failed` | Server->Client | Deployment failed |

## 8. Acceptance Criteria Summary

### 8.1 Must Have (P0)
1. Users can view all three layers with current status
2. Users can deploy a layer to any environment
3. Real-time deployment logs are visible
4. Deployment respects layer dependencies
5. Users can rollback a deployed layer

### 8.2 Should Have (P1)
1. Interactive dependency graph
2. Configuration editing with validation
3. Environment-specific overrides
4. Cost estimates per layer

### 8.3 Nice to Have (P2)
1. Deployment timeline with filtering
2. Version comparison
3. Animated deployment flow
4. Export deployment history

## 9. Constraints and Assumptions

### 9.1 Constraints
- Must integrate with existing DesignWizardContext
