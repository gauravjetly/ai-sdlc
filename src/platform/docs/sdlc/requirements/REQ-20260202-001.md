## 1. Executive Summary

### 1.1 Problem Statement
The Infrastructure Designer currently has placeholder content in the wizard drawer. Users need a guided, step-by-step experience to design AWS network infrastructure that follows best practices, validates configurations in real-time, and generates deployable Terraform code.

### 1.2 Business Objectives
- **Reduce Design Time**: From hours to minutes for standard network architectures
- **Ensure Compliance**: Built-in AWS best practices and security standards
- **Eliminate Errors**: Real-time validation prevents misconfigurations
- **Enable Reuse**: Save designs as templates for future projects

### 1.3 Success Metrics
- 90% of users complete wizard without errors on first attempt
- Network design time reduced by 70%
- Zero security misconfigurations in generated infrastructure
- 85%+ test coverage on all components

## 5. Data Structures

### 5.1 VPC Configuration
```typescript
interface VPCConfig {
  name: string;
  cidrBlock: string;           // e.g., "10.0.0.0/16"
  enableDnsSupport: boolean;   // default: true
  enableDnsHostnames: boolean; // default: true
  enableIpv6: boolean;         // default: false
  tags: Array<{key: string; value: string}>;
}
```

### 5.2 Subnet Configuration
```typescript
interface SubnetConfig {
  id: string;                  // UUID for tracking
  name: string;
  cidrBlock: string;           // e.g., "10.0.1.0/24"
  availabilityZone: string;    // e.g., "us-east-1a"
  isPublic: boolean;
  mapPublicIpOnLaunch: boolean;
  tags: Array<{key: string; value: string}>;
}
```

### 5.3 Routing Configuration
```typescript
interface RoutingConfig {
  internetGateway: {
    enabled: boolean;
    name: string;
  };
  natGateways: Array<{
    id: string;
    name: string;
    subnetId: string;          // Public subnet for NAT placement
    eipAllocationId?: string;
  }>;
  routeTables: Array<{
    id: string;
    name: string;
    isMain: boolean;
    routes: Array<{
      destinationCidr: string;
      targetType: 'igw' | 'nat' | 'local' | 'vpce';
      targetId: string;
    }>;
    subnetAssociations: string[]; // Subnet IDs
  }>;
}
```

### 5.4 Security Group Configuration
```typescript
interface SecurityGroupConfig {
  id: string;
  name: string;
  description: string;
  ingressRules: Array<{
    id: string;
    protocol: 'tcp' | 'udp' | 'icmp' | '-1';
    fromPort: number;
    toPort: number;
    sourceType: 'cidr' | 'sg' | 'prefix-list';
    source: string;            // CIDR or SG ID
    description?: string;
  }>;
  egressRules: Array<{
    id: string;
    protocol: 'tcp' | 'udp' | 'icmp' | '-1';
    fromPort: number;
    toPort: number;
    destinationType: 'cidr' | 'sg' | 'prefix-list';
    destination: string;
    description?: string;
  }>;
  tags: Array<{key: string; value: string}>;
}
```

### 5.5 Network Layer Data (Complete)
```typescript
interface NetworkLayerData {
  vpc: VPCConfig;
  subnets: SubnetConfig[];
  routing: RoutingConfig;
  securityGroups: SecurityGroupConfig[];
  validationResult?: {
    isValid: boolean;
    errors: ValidationError[];
    warnings: ValidationError[];
    info: ValidationError[];
  };
}
```

## 6. User Flows

### 6.1 Happy Path: Complete Network Design

```
1. User clicks "Configure" on Network Layer card
2. Wizard drawer opens on VPC Configuration step
3. User enters VPC name: "production-vpc"
4. User selects CIDR: 10.0.0.0/16
5. User enables DNS resolution (default) and DNS hostnames
6. User clicks "Next"
7. Wizard advances to Subnet Design step
8. User adds public subnet: "public-1a" in us-east-1a with 10.0.0.0/24
9. User adds public subnet: "public-1b" in us-east-1b with 10.0.1.0/24
10. User adds private subnet: "private-1a" in us-east-1a with 10.0.10.0/24
11. User adds private subnet: "private-1b" in us-east-1b with 10.0.11.0/24
12. User clicks "Next"
13. Wizard advances to Routing Configuration step
14. User enables Internet Gateway: "production-igw"
15. User enables NAT Gateway in public-1a
16. Route tables are auto-configured
17. User clicks "Next"
18. Wizard advances to Security Groups step
19. User creates "web-sg" with rules: HTTP (80), HTTPS (443) from 0.0.0.0/0
20. User creates "app-sg" with rules: 8080 from web-sg
21. User creates "db-sg" with rules: 5432 from app-sg
22. User clicks "Next"
23. Wizard advances to Network Validation step
24. Validation runs automatically
25. All checks pass (green checkmarks)
26. User clicks "Save"
27. Configuration is persisted
28. User clicks "Deploy"
29. Terraform is generated and applied
30. Network layer status changes to "Deployed"
```

### 6.2 Error Flow: Invalid CIDR

```
1. User is on VPC Configuration step
2. User selects "Custom" CIDR
3. User enters "192.0.2.0/24" (public TEST-NET range)
4. Field shows error: "CIDR must be private (RFC 1918): 10.x, 172.16-31.x, or 192.168.x"
5. "Next" button is disabled
6. User corrects to "192.168.0.0/16"
7. Error clears
8. "Next" button enables
```

### 6.3 Error Flow: Overlapping Subnets

```
1. User is on Subnet Design step
2. User has subnet: 10.0.0.0/24
3. User tries to add subnet: 10.0.0.128/25
4. Error appears: "Subnet CIDR 10.0.0.128/25 overlaps with 10.0.0.0/24"
5. User cannot save overlapping subnet
6. User changes to 10.0.1.0/24
7. Subnet is added successfully
```

## 7. Integration Points

### 7.1 DesignWizardContext
- **updateLayerData**: Store network configuration
- **addNode**: Add VPC, subnet, gateway nodes to canvas
- **validateCurrentLayer**: Trigger network validation
- **saveProgress**: Persist to backend API

### 7.2 Backend API Endpoints
- `PUT /api/v1/workflows/{id}`: Save workflow progress
- `POST /api/v1/workflows/{id}/validate`: Run validation
- `POST /api/v1/workflows/{id}/deploy`: Deploy to AWS

### 7.3 ReactFlow Canvas
- Generate nodes for: VPC (container), Subnets (grouped by AZ), IGW, NAT, Route Tables
- Generate edges for: Subnet-to-RouteTable associations, RouteTable-to-Gateway routes
